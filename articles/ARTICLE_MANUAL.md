# 大阪歯科総研 記事執筆マニュアル（2026-07-08 制定）

このマニュアルは、2026-07-06〜07-08にかけてユーザーとのやり取りで確立した
「大阪歯科総研の記事はこうあるべき」というルールを一つにまとめたものです。
**今後、日次で新規記事を生成する際は、必ずこのマニュアルの基準を満たすこと。**

新規記事は `~/Desktop/クロード/AI評判設計システム/blog_generator.py`（SITE_TYPE=portal）で
生成されます。このマニュアルのルールは、同スクリプトのプロンプト・`to_html()`に
2026-07-08時点で反映済みです。プロンプトやテンプレートを変更する場合は、
**このマニュアルとの整合性を必ず確認すること**。

## 0. コラムは全都市サイト共通・都市名禁止（2026-07-10 制定・最優先ルール）

- コラムは**全ブランド共通の1か所に集約**する：歯科総研は `shikasoken.com/articles/`、
  動物ナビは `osaka.doubutsu-navi.com/articles/`。他都市サイトの `articles/index.html` は
  共通コラムへのリダイレクトページ（都市サイト側で `build_index.py` を実行してリダイレクトを
  上書きしないこと）
- **記事タイトル・見出し・本文に都市名・区名・地名を入れない**（どの都市の読者が読んでも
  成立する内容にする）。実データ統計は「当サイトで◯院を分析したところ」と表現する
- 2026-07-10方針確定：既存の大阪記事27本は**削除せず、本文・タイトルから地名を除去して存置**
  （ファイル名＝URLはSEO維持のため旧のまま。ファイル名の「大阪」は許容）。`build_features.py` の
  CATEGORY_ARTICLES は地名除去後のタイトルで再登録済み（表示タイトルとファイル名が異なる点に注意）
- 毎日投稿はブランドごとに1系統のみ（歯科＝大阪リポジトリで生成、動物ナビ＝大阪ペット医療ナビ）。
  やらないことリスト第2項（記事投稿の都市展開凍結）と整合
- このルールは `blog_generator.py`・`keyword_survey.py`・`client_config*.json`（_column_policy）に
  2026-07-10反映済み。プロンプトを変更する場合は本節との整合を必ず確認すること
- **2バケット原則（2026-07-13 制定）**：コンテンツは「普遍か・地域固有か」で扱いを分ける。
  ①**一般コラム（普遍）**＝歯周病・インプラント等の知識記事 → 本節のとおり共通1か所に集約・
  都市サイトはリダイレクト維持。②**研究記事（地域固有・`articles/research/`）＝共通集約の対象外**。
  各都市の `clinic_db.json` から実データを算出する本質的に都市固有のコンテンツなので、
  **各都市サイトが自前の `articles/research/` を持ち、その都市サイト内のリダイレクトされない面**
  （医院ページ・sitemap・トップのフッター「データ研究」）から自都市の研究へリンクする。
  研究記事のみ、統計の文脈で都市名を使ってよい（タイトルは都市名なしの共通ルール維持）。
  生成は `build_data_report.py`（都市非依存・実行は都市ごと）。**都市サイトで `build_index.py` を
  実行しない原則はこれまでどおり**（研究導線に build_index.py は不要）

## 1. 記事の中心コンセプト

大阪歯科総研の記事は、一般的なSEO記事ではない。

> 「歯医者に直接聞きづらいけど、実は知りたい問いに答える記事」

読者が読み終えたときに「これは知りたかった」「これなら歯医者で聞けそう」
「他の歯科サイトより役に立つ」と思えることがゴール。

記事の中心に置くべき問いの例：
- これって本当に歯医者に行くレベル？
- 何日くらいなら様子を見てもいい？
- この症状は普通？それとも危ない？
- 治療費はいくらくらい覚悟すればいい？
- 保険でできる？自費をすすめられたらどう判断する？
- 先生に何を聞けばいい？
- 別の医院に相談しても失礼じゃない？
- 治療を断ってもいい？
- 今の医院で続けていいか、変えた方がいいか？

## 1.5 読者を離さない書き方（2026-07-08追加・毎日投稿の品質基準）

「内容が薄い」「文章ばかりで読み疲れる」への対策。blog_generator.pyのプロンプトにも
同内容を反映済み（片方だけ直さないこと）。

1. **タイトルは「クリックされる型」から選んで作る**（毎回同じ型に偏らせない）
   - ①実データ数字型：「夜間対応の案内がある歯科は約6%」（当サイト最強の型・都市名は付けない）
   - ②本音の代弁型：「歯医者を変えたい。でも言い出しにくい人へ」
   - ③意外な事実・逆説型：「口コミ4.5の医院が、あなたに合うとは限らない理由」
   - ④場面切り取り型：「詰め物が取れた夜、まず何をすればいいか」
   - ⑤判断基準型：「様子見していい痛み・今夜連絡すべき痛みの分かれ目」
   - ⑥損失回避型（控えめに）：「知らずに損しがちな、保険でできる治療の範囲」
   - 禁止：釣りタイトル（本文が答えない問い）・恐怖の誇張・「衝撃」「ヤバい」等の
     下品な釣り語・医療広告規制に触れる断定/最上級。**タイトルで立てた問いには
     本文の最初のセクションで必ず答える（釣り逃げ禁止）**
   - 導入は読者の内心の代弁か実データの意外な数字で。「〜しましょう」
     「〜が大切です」で始まる導入は禁止（一般論の合図）
2. **当サイト独自の実データを最低1〜2回織り込む**
   - `blog_generator.py`の`compute_db_stats_text()`がclinic_db.jsonから実数統計
     （夜間診療◯院・駐車場◯院・駅500m以内◯院など）を自動算出してプロンプトに渡す
   - この実数は他メディアには書けない当サイトだけの武器。**提供された数字以外を
     創作することは絶対禁止**（禁止事項2-2と表裏一体）
   - 表現は必ず「案内が確認できた医院は◯院」（断定しない）
3. **3セクションに1回は視覚要素（compare/checklist）を挟む**：文章の壁を作らない
4. **各セクションの1文目で結論**：流し読みでも意味が通る
5. **回遊の種をまく**：まとめ直前で関連する別の疑問に1行触れる（記事CTAの
   条件付きランキングリンクと、サイドバーの関連記事が受け皿になる）

## 1.6 確定した文章ドクトリン（2026-07-13制定・到達品質の基準）

サンプル2本＝`2026-07-13_初診のレントゲン・検査代、3割負担で何が変わるのか.html` と
`2026-07-13_歯医者で泣き叫ぶ我が子、親が本当にすべきこと.html` が**到達品質の見本**。
以後の全記事・全リライトはこの2本の水準を満たす。ルールは blog_generator.py の
プロンプト／to_html／keyword_survey.py にも反映済み（片方だけ直さないこと）。

1. **患者感情ファースト**：書き出しは読者の感情・不安・気後れに寄り添う。
   「まず、肩の荷を下ろしてほしい」「これは、あまり知られていない」のように内心へ一歩
   踏み込んでから本題へ。「〜しましょう」「〜が大切です」の一般論の入り方は禁止（§1.5と同じ）。
2. **当たり前を書かない・本文に混ぜない**：「早めの受診が大切」等の誰でも知っている一般論は、
   1文でも本文の主役にしない。前提として要るときも、素早く非自明な話（下記3）へ橋渡しする。
3. **非自明な「明日使えるネタ」＋根拠を必ずセットで**：読者が明日その足で使える意外な具体策を
   各記事に最低1つ（例：費用は診察室でなく予約時・受付で聞く／中断1か月超で初診料が再びかかる／
   「痛くないよ」は逆効果）。**必ず根拠も明記**（診療報酬でこう決まっている／なぜ逆効果か）。
   根拠を示せない裏技は書かない（§2-4の捏造禁止と表裏一体）。
4. **デリケートな話はオブラートに**：費用・お金・叱責・失敗・恥ずかしさに触れる話題は、読者を
   責めず追い詰めない柔らかい表現で。「損している」→「実は一番損をしがち」、気持ちを認めてから解く。
5. **短い段落・行間2.0・要点は箱**：1段落は1〜2文で改段（3文以上詰めない）。セクションの入りは
   1文だけの短い段落でよい。要点・手順・言い換えの対比は地の文に流さず箱（要約の①②③・
   question-list・compare/checklist・表）に入れる。本文の行間は `.art-body p{line-height:2;margin:0 0 1.15em;}`
   で広く組む（blog_generator.pyのto_htmlが新規記事へ自動付与。既存記事へ手当てする場合はこの1行を
   記事内`<style>`へ追加）。

## 2. 絶対禁止事項

以下は**理由の如何を問わず禁止**。1つでも該当したら公開前に修正すること。

### 2-1. 抽象語・当たり前の一般論だけで終わらせること
次の表現は、それ単体で終わらせてはいけない。必ず「何を確認すれば判断できるか」まで
具体化する。

| 禁止（抽象語） | 良い例（具体化） |
|---|---|
| 口コミの質を見る | 「先生が優しい」だけでなく、治療前に選択肢を説明してくれたか・費用を先に説明してくれたか・痛みが出た時の対応があったか・無理に自費をすすめられなかったか、を見る |
| 説明が丁寧 | 現在の状態・治療の選択肢・費用・回数・リスクを説明してくれるか |
| 専門医がいると安心 | 自分の症状がその専門性を必要とするケースかを見る（例：根管治療なら再治療・難治症例・マイクロスコープ使用の有無） |
| 設備が整っている | その設備が今回の治療に本当に必要か、無くても対応できるケースはあるか |
| 成功率が高い | 数字そのものではなく、精度に関わる体制（マイクロスコープ・ラバーダム・CT撮影・再治療対応）を確認する |
| 通いやすい | 何を基準に通いやすいと判断するか（診療時間・当日対応・立地） |
| 信頼できる | 何を確認すれば信頼できると判断できるか |

### 2-2. 答えられない問いに無理に答えない
悪い例：「成功率は公式サイトや口コミで確認しましょう」
→ 患者が正確な成功率を事前に確認するのは現実的に難しい。
良い例：「医院ごとの正確な成功率を患者が事前に確認するのは難しいのが実情です。
その代わり、成功率に関わりやすい要素（マイクロスコープの使用、ラバーダム防湿、
CT撮影の有無、再治療症例への対応、治療前説明の具体性）を確認しましょう」

### 2-3. 数字を勝手に基準にしない
悪い例：「初診時に20分以上の説明があると安心です」
良い例：「説明時間の長さそのものより、現在の状態・治療の選択肢・費用・回数・リスクを
説明してくれるかが重要です」

### 2-4. 根拠のない数値・煽り表現・医療効果の断定
- 「虫歯リスクが3倍」「再治療費用3,000〜8万円」のような断定的な数字・相場は書かない
- 費用は「〇〜〇円」と断定せず、「何が費用を左右するか」「治療前に何を確認すべきか」を書く
- 「必ず治る」「改善します」など医療効果を断定する表現は禁止
- ランキング表現（1位・〇〇No.1）は禁止

### 2-5. 同じ見出し・同じ構成の使い回し
全記事で同じ見出し文言をコピーしない。記事タイトルに応じて、見出し自体を変える。

### 2-6. 都市名・区名・地名を入れる
コラムは全都市サイト共通のため、地域を特定する表現は一切使わない（§0参照）。

## 3. 記事の構成（8ブロック）

1. **結論**（AI Summary）：不安に寄り添う1文＋結論1〜2文。断定しすぎない
2. **この記事で分かること**（covers・サイドバー表示）：本文セクションと同じ個数・同じ順序の短い句
3. **本文セクション**（sections）：6〜8個。見出しは記事タイトル固有の、患者の本音に近い言葉
4. **判断に迷う分岐**：本当に2択の比較になる内容の時だけ、セクション内に compare を使う
5. **患者が歯科医師に確認したいこと**（question_section）：後述（§4）
6. **費用・回数・自費提案で確認すること**：セクションの一つとして扱う（専用ブロックにしない）
7. **医院選びで見るべき具体項目**（clinic_points）：3〜4個
8. **FAQ**：5個、「聞きづらい」系の質問
9. **AI診断への導線**（clinic-guide）：記事末尾の固定CTA

各セクション本文（paragraphs）には、必ず次の3点を含めること。
①結論　②なぜそう言えるか　③患者が実際に確認できる質問・見るべき点
単なる感想・一般論の羅列で終わらせない。

## 4. 「患者が歯科医師に確認したいこと」セクションのルール

このセクションは**全記事に必須**。ただし過去に2つの重大な問題があったため、
必ず以下を守ること。

### 4-1. 質問の主語は必ず「患者」
患者が歯科医師に聞く質問にする。**歯科医師が患者に確認する問診項目を絶対に入れない。**

| 禁止（医院側の問診） | 良い例（患者が医師に聞く質問） |
|---|---|
| いつから痛いですか？ | この痛みは、他の原因も考えられますか？ |
| どのくらい痛いですか？ | この症状は今日中に診てもらうべきですか？ |
| どこが痛いですか？ | この治療は専門的な対応が必要なケースですか？必要な場合、専門医や口腔外科へ紹介してもらえますか？ |
| 治療に関する説明は具体的ですか？ | 治療内容・費用・通院回数について、どこまで事前に説明してもらえますか？ |
| 過去に同じ症状はありましたか？ | （使わない：これは問診項目そのもの） |

書いたら必ず自己チェックする：「これは患者が歯科医に聞く質問か？（医院が患者に
聞く問診になっていないか）」

### 4-2. 「はい/いいえ」で終わる薄い質問にしない
悪い例：「専門医はいますか？」
良い例：「この治療は専門的な対応が必要なケースですか？必要な場合、専門医や口腔外科へ
紹介してもらえますか？」

### 4-3. 見出しに「聞いていい質問」という言葉を使わない
理由：幼く見える／記事全体の高級感を下げる／研究機関的なトーンに合わない／
全記事で同じ見出しになるとテンプレ感が強くなる。

記事テーマに応じて、以下のように**記事ごとに異なる**見出しにする（同じ見出しの
使い回し禁止）：
- 受診前に確認したいこと
- 診察前に整理しておきたい質問
- 治療を決める前に確認したいこと
- 費用を決める前に確認したいこと
- 相談前に整理しておきたい論点
- 初診前に見ておきたいポイント

### 4-4. セクションは1記事に1つだけ
本文の各セクションに`questions`をばらまかない。専用セクション1つに5〜8個の
質問を厳選してまとめる（過去に1記事へ5〜6個のボックスが乱立し「幼稚」
「テンプレ感」の原因になったことがある）。

### 4-5. 導入文は固定文言
「診察時に聞きづらいことほど、先に整理しておくと相談しやすくなります。
気になるものをメモして、受診時に確認してみてください。」

## 5. サイドバー「受診前に整理したいこと」（prep_note）のルール

`question_section`（患者→医師の質問）とは**別物**。こちらは症状・事実ベースの
短いチェック項目で、患者が予約前に自分の状況を整理するためのもの。質問文ではなく
短い句にする。

例：
- 詰め物記事：予約時に伝えること → いつ取れたか／痛みがあるか／しみるか／取れた物を持っているか
- 顎関節症記事：受診前に整理したいこと → いつから痛いか／口が開きにくいか／噛むと痛いか／音が鳴るか／歯ぎしりの自覚があるか

## 6. デザイン方針

「カードを並べたWebフォーム」ではなく「余白のある編集記事」にする。

- **カードを増やして整理するのではなく、情報の強弱をつける**
- カードとして扱ってよいのは **AI Summary** と **記事末尾のCTA（clinic-guide）** だけ
- 見出し（h2）に緑の縦線装飾を使いすぎない。太字テキストのみで十分
- 本文中の2択比較は、カード化せず `.compare-thin`（細い罫線で区切るだけ）を使う
- 注意が必要な時だけ `.mini-checklist`（小さなチェックリスト）を使う。多用しない
- 大きな白いカードを連続させない
- 箇条書きだけで終わらせない。前後に「なぜ重要か」を1〜2文添える
- 余白は広く、でもスカスカにしない
- ヒーローに装飾用のダミー画像・アイコン・パネルを追加しない（過去に何度か
  追加しては差し戻された。テキストのみのヒーローで十分）

### レイアウト（PC表示）
- コンテナ幅：1120px（`--container-article`）。本文の読みやすさは
  `.art-main{max-width:820px}`で別途担保
- 900px超で2カラム：左＝`.art-main`（タイトル・リード・AI Summary・本文・FAQ・CTA）、
  右＝`.art-side`（この記事で確認すること／受診前に整理したいこと／AI診断への
  小CTA／関連記事／注意書き）
- `.art-side`はオフホワイト・薄い枠線・影なし・`position:sticky`
- 900px以下は1カラムに戻り、サイドバーは本文の下に自然に移動
- サイドバーの「この記事で確認すること」の各項目は、本文の対応セクションへ
  スムーズスクロールするリンクにする（`<h2 id="sec-N">`と対応）。**この時、
  covers配列はsections配列と同じ個数・同じ順序で生成すること**（ズレると
  誤動作する）
- アンカーリンクは素の`<a href="#...">`のままにせず、クリックを`preventDefault()`
  してから`scrollIntoView()`＋`history.replaceState()`で処理する（ブラウザの
  「戻る」ボタンが1回で前ページに戻れるようにするため。素のハッシュリンクだと
  クリックのたびに履歴が積まれ、スクロール回数分「戻る」を押す必要が出るバグになる）

## 7. 記事サムネイル写真

- 用途は**記事一覧ページ（`articles/index.html`・`cat-*.html`）のカードサムネイルのみ**。
  記事ページ本体のヒーローや本文には使わない
- 写真は`サムネイル画像/`配下に症状別カテゴリで用意されている（各カテゴリ16枚）

## 9. 実装上の注意点（スクリプトを触る人向け）

過去に何度か発生したバグとその対策。今後スクリプトを修正・追加する際に再発させないこと。

- **covers⇔sectionsの対応付け**：新規記事は`blog_generator.py`が生成時点で`covers[i]`と
  `sections[i]`を同じ順序にするため、`id="sec-{i+1}"`への機械的な対応で正しく動く。
  一方、**既存記事を後から手直しする場合**（covers と本文見出しの言い回しが元々ズレている
  ことがある）は、単純な位置対応ではなく、文字列の類似度でマッチングし直す必要がある。
  `difflib`の`SequenceMatcher`は日本語の短い句同士では精度が低かったため、
  **文字バイグラム（2文字の組）のJaccard類似度**を使うこと（`link_covers_to_sections.py`参照）
- **`.art-body`の終端検出**：`.art-body`は内部に`.compare-thin`/`.mini-checklist`/
  `.question-list`等、入れ子の`<div>...</div>`を複数含むため、非貪欲正規表現
  `<div class="art-body">(.*?)</div>`で終端を取ろうとすると、最初の入れ子divの
  `</div>`で止まってしまい本文の大部分を取りこぼす。**`<div class="faq">`の直前までを
  art-bodyの終端とみなす**方式で実装すること（テンプレート上、常にこの順序で並ぶ前提）
- **ページ内アンカーリンクはブラウザ履歴に注意**：素の`<a href="#sec-N">`をそのまま
  クリックさせると、クリックのたびに履歴エントリが積まれ、「戻る」を複数回押さないと
  前のページに戻れなくなる（例：5回スクロール→6回目でようやく前ページに戻る）。
  必ず`preventDefault()`→`scrollIntoView()`→`history.replaceState()`のパターンで実装する
  （`blog_generator.py`の生成テンプレート・全既存記事の`<script>`タグに実装済み）
- **記事本文を複数のスクリプトで段階的に上書きする場合**：過去に、詰め物・親知らず・
  セカンドオピニオンの3記事で見出し＋導入文が二重に挿入されるバグが発生した（パッチ工程が
  複数回に分かれたことが原因）。**修正後は必ず全記事で`<h2>`の重複を`Counter`でチェック**
  すること（§8のコマンド例を使う）
- **重複記事の防止（2026-07-09追加）**：毎日投稿では「一度書いた記事と同じような記事」の
  再生成を二段構えで防ぐ。①`keyword_survey.py`が公開済み記事タイトル一覧（articles/の
  ファイル名から取得）をテーマ提案プロンプトに渡し「同じ疑問に答えるテーマは提案禁止・
  本数より新規性優先」と指示。②`blog_generator.py`の`check_duplicate_theme()`が生成前に
  チェック（バイグラム類似度0.5以上→即スキップ／それ以外→Haikuで意味判定・1回1円未満）。
  重複時はexitコード2で終了し、`daily_post.sh`が次の候補テーマに自動で切り替える。
  手動で強制生成したい場合は`--force`。
  **候補テーマは6件**（2026-07-10に3件→6件へ増量）：記事が27本を超えた頃、候補3件が全て
  既存記事と意味重複して「本日の投稿0本」になる事故が起きた。候補を増やしても
  「新規性優先・無理な埋め合わせ禁止」の指示は維持している。全候補が重複でスキップされた
  日は投稿なしが正常動作（無理に類似記事を量産しない）。
  **注意**：バイグラム類似度は日本語の言い換え（「夜間に歯が痛い時の探し方」vs
  「歯が痛い時すぐ診てもらえる歯医者選び」＝スコア0.06）を検出できないと実測で判明済み。
  機械判定だけに頼る改修をしないこと（AI意味判定が本体）
- **見出しの改行対策**：記事タイトルに「Ａ｜Ｂ」「Ａ？大阪Ｂ」「Ａ！大阪Ｂ」のような区切りが
  あると、ブラウザが区切り記号の直後で改行し、1〜2文字だけが浮いた不自然な見た目になる。
  `break_title_at_separator()`（`blog_generator.py`）で`<h1>`に確実な`<br>`を挿入して対応済み
  （`<title>`やalt属性などHTMLタグを埋め込めない箇所には使わない）。新しい区切りパターンが
  見つかった場合は、この関数と大阪歯科総研側の`nowrap_pipe()`（`build_index.py`等）の
  **両方**を直すこと

## §10 記事の信頼パッケージ（2026-07-10 制定・自動付与）

全記事の末尾（まとめの後）に「この記事について」ボックスが自動で付く（blog_generator.pyのto_html）。
内容：組織著者「{サイト名} 編集部」・公開日・根拠（自社データ）・編集方針(policy.html#editorial)への
リンク・訂正窓口。あわせてArticle JSON-LD（author=Organization・datePublished）がheadに入る。

**理由**：AIっぽさの正体は「誰が・いつ・何を根拠に書いたか」の欠落。YMYL（医療）では特に、
これらの開示がE-E-A-T（信頼）の土台になる。

**禁止事項（最重要）**：
- 偽の人間著者・架空の監修者・実在しない肩書きを作らない（発覚時に全信頼が崩壊する）
- AI利用を隠さない。編集方針に「AIによる下書きと分析を活用し、編集部が確認」と開示している
- この開示方針と矛盾する文言（「専門医が執筆」等）を記事内に書かない

記事を手動修正するときも信頼ボックスは削除しない。日付は公開日を維持し、
大きく書き直した場合はdateModifiedのみ更新する。

## 12. 開業医向けデータ研究記事（別読者・pro research・2026-07-14 制定）

患者向けコラム（§0〜§11）とは**読者もルールも別**の記事系統。読者＝歯科医院の院長・歯科衛生士。
狙い＝「大阪歯科総研だから書ける、医院経営者向けのデータ研究」で信頼を作り、自院を振り返らせ、
医院向けの相談（AI評判設計）へ自然につなぐ。週1本を**半自動**（私が生成→ユーザー承認→公開）で運用。

### 生成の場所（患者記事とは別エンジン）
- 実体は `build_data_report.py` の `_SERIES_BUILDERS` に追加するビルダー関数。dict に
  `audience="clinic"` / `eyebrow`（FOR CLINICS表示）/ `cta`（医院向けページへ）を持たせる。
  研究トップ（articles/research/）のシリーズ棚に「開業医向け」バッジ付きで並ぶ。
- **blog_generator.py / keyword_survey.py は触らない**（あれは患者の毎日投稿用）。よって§7の
  「記事ルール変更の同期チェックリスト」の対象外。pro記事の同期先は
  ARTICLE_MANUAL.md（本節）＋build_data_report.py＋CLAUDE.md変更履歴＋メモリ。

### 中身のルール（患者記事と違う点）
1. **都市名は使ってよい**（§0/§2-6の都市名禁止は患者コラム＝全都市共通のためのルール。研究記事は
   「大阪市の市場」を語るのが主題なので `CITY`＝大阪市を明示する。他都市展開時は各都市の値になる）。
2. **数字は必ずビルド時にDBから算出（捏造ゼロ・月次自動更新）**。前提が崩れた発見は自動で伏せる。
3. **相関を出すときは因果と峻別**を本文・FAQ・方法論に。さらに逆の因果（人気→サイト充実）や共通要因の
   可能性まで踏み込む。「◯◯すれば集患が増える」と実測で示していない効果を断定しない
   （例：「書けば効きます」→「便益を添えて書く余地があります」）。
4. **所有率と訴求率を区別**。サイト解析由来の指標は「持っている」でなく「サイトで訴求している」と書き、
   「未記載＝未保有ではない」と明記する。
5. **方法論に統計の前提を必ず開示**：母数（掲載院／解析できた院）・平均はGoogle評価のある院のみ・
   口コミ中央値は1件以上の院で算出（0件除外）・区別は各区25院以上・**有意差検定は行わず記述統計上の傾向**。
6. **図表で"呼吸"を作る**：純CSSの横棒（`_bars`）・比較（`.rp-compare`）・表（`.rp-table`）・
   チェック（`.rp-check`）。地域データは必ず区別の横棒で視覚化し、読者が自院の区を探せるようにする。
7. **主題がぼやけないよう冒頭に道しるべ**（この記事で確かめる問いを2〜3点）。分量が多くても軸を先に示す。
8. **文体は§1.6のドクトリンを職業人向けに転用**：院長の内心に寄り添う入り（設備投資した心理等）→データ→
   実務に翻訳（明日サイトで直せること）。当たり前で終わらせない。短段落・要点は箱。

### 商談への導線（やってよい／やってはいけない）
- **やってよい**：記事末尾に**静的な自院チェック**（□項目・送信なし・PIIなし・自己採点の目安）を置き、
  その後に医院向けページ（shikumi.html）への具体的CTA（同区データと照らして見直す等）を1つ。

  （大きな新機能）・個人情報/申込の絶対禁止に抵触**。営業解禁（GA4実数がレポートに載る）後に別途検討。
